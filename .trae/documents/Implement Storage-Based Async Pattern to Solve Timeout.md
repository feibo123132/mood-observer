To solve the "Super Tricky" problem (504 Gateway Timeout & CORS), I will implement a robust **Storage-Based Async Pattern**. This bypasses the Gateway's hard 60s timeout limit by allowing the client to "resume" or "retry" and pick up the result generated by the background process.

### **Core Concept**
1.  **Backend**: Instead of returning the audio file directly (which times out), the Cloud Function will upload it to Cloud Storage and return a `fileID`.
2.  **Idempotency**: Before generating, the function checks if the file `reports/podcast_{year}_{week}.mp3` already exists. If yes, it returns the existing file immediately.
3.  **Frontend**: If the request times out (504), the UI will guide the user to "Retry". The second attempt (Retry) will find the file generated by the first attempt (which continued running in the background) and succeed instantly.

### **Detailed Implementation Plan**

#### **1. Backend: `cloudfunctions/generatePodcast/index.js`**
-   **Initialize Cloud Storage**: Use `app.uploadFile` and `app.getTempFileURL`.
-   **Input**: Accept `year` and `week` in addition to `text`.
-   **Step A: Check Existence**:
    -   Construct path: `reports/podcast_${year}_${week}.mp3`.
    -   Check if file exists (using `getTempFileURL` as a proxy check, or `deleteFile` check, or DB lookup). *Simplest robust way*: Try to get temp URL. If valid, return it.
-   **Step B: Generation & Upload**:
    -   Perform the existing WebSocket generation.
    -   **Upload** the buffer to Cloud Storage (`cloud.uploadFile`).
    -   Get the `fileID` and then the `tempFileURL`.
    -   Return `{ success: true, audioUrl: tempFileURL, fileID }`.

#### **2. Frontend: `src/pages/HistoryReportsPage.tsx`**
-   **Update Call**: Pass `year` and `week` to `callFunction`.
-   **Error Handling**:
    -   If error contains `504` or `timeout`:
        -   Change UI state to a "Generating in background..." mode.
        -   Show a **"Check Status / Retry"** button instead of "Generate".
        -   Explain to user: "The report is being generated in the background. Please click Retry in a moment."
    -   *Better UX*: Auto-retry once after 10 seconds? No, manual retry is safer for "super tricky" debugging.

#### **3. Configuration: `cloudbaserc.json`**
-   Create this file to ensure CLI/Local tools know the environment ID (`jieyou-3gr01mvob9ad92de`).

### **Why this solves it:**
-   **504 Timeout**: We accept that the *first* request might die at 60s. But the *function* (timeout 300s) keeps running.
-   **CORS**: Often a side-effect of the 504. If the retry returns fast (<1s), it won't timeout and won't trigger the 504-induced CORS error.
